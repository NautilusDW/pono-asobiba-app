「保存できない」「動かない」系のユーザ主訴は、**そのまま** バグ仮説に翻訳しない。

## ルール
ユーザが「X ができない」と言っても、X はユーザ視点の表面症状で、内部では **save 自体は成功 / apply 側の欠陥 / そもそもの仕様欠落** など複数の原因がある。最初に実機で「何が実際に起きているか」を切り分ける。

**Why:** 2026-05-07 の Quizland ステージ絵問題で、ユーザの「全然保存できない」という訴えに対して前任セッションは **保存処理自体のバグ** だと推測してずっと探索したが、実際は:
1. 保存は完璧に動いていた (PUT 200, JSON 更新確認)
2. apply 側でも巻き戻しは起きていなかった
3. 真の課題は **per-question 機構が元から実装されていなかった** こと（全 169 問が単一キーを共有していた）

ユーザが過去発言で「フクロウは共通でいいけど、ステージングの絵は 1 問ごとに」と明示していたのを見落とすと、何時間も別の方向に走る。

**How to apply:**
- ユーザが「動かない」と言ったら、最初に「どう動かして、どう動いてほしいのに、実際何が見えるか」を 1 つの実機テストで切り分ける（推測の羅列を避ける）
- ユーザの過去発言を引き継ぎ文書から逐語的に拾い、表面症状の裏にある **本当の要求** を抽出する
- 「直前の修正が効いてない」という思い込みを疑う前に、**そもそも対象機構が実装されているか** を確認する
- 引き継ぎ文書の「事実」セクションも疑う（前任が「これが事実」と書いた `.emoji-display|0` は全問題共有 が、実は直すべきバグそのものだった）
