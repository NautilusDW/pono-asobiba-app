共通して発覚した設計上の落とし穴 (全案の実装前チェックリストに含めること):

1. **`_resolveQuiz` / `onPick` の流用は要注意**: 迷路の `_resolveQuiz` はクイズ4種専用で、リトライ無制限型ゲームには合わない (water_bridge / donguri_bowl は独自 resolve + `_closeEncounter(true)` 直呼びが正しい前例)。なぞなぞ側も onPick の失敗パスを迂回すると星評価・noMissボーナスが壊れる。
2. **登録テーブルの多点同期**: 迷路は `_GAME_LABELS` / `_GAME_DIALOGS` / `_GAME_STARTERS` / `MAZE_ADMIN_DEBUG_GAMES` / `_setEncounterMode` / `_closeEncounter` / 回帰テストの7〜8箇所、なぞなぞは showQuiz / buildQList / クリーンアップ4〜5箇所への追加漏れが静かな不具合になる。整合性を自動検証するテストの先行導入を推奨。
3. **iOS タッチ対策の正解は Pointer Events + pointercancel + touch-action:none**: 複数案の原文が memory の教訓を「touch イベントに切り替えよ」と誤読していた。両ゲームの既存ドラッグ実装が確立した規約に統一する。
4. **「詰み」の排除**: 運動・記憶課題はクイズと違い「親に聞けば必ず解ける」救済経路が無い。全案で救済 (pity / 再表示 / ヒント誘導) の明文化が必須。
5. sw.js CACHE_VERSION バンプ、画像AR厳守、AGENTS_CLAIMS.md での claim (nazonazo 回帰テストは現在進行形で編集中の並走差分あり) を各案共通の出荷条件とする。

## 第1章 迷路 (maze/)

迷路のエンカウンターは 9 種のミニゲームが同一ディスパッチテーブルを共有するテーブル駆動設計で、10個目以降の追加自体は構造的に安全。問題は各案の「勝敗・リトライの状態設計」と「対象年齢下限 (3歳) の詰み防止」に集中している。

### 1-1. はっぱめくり ペアさがし (神経衰弱ミニ) — 推奨度: すぐ着手 (修正反映後)

- **スコア**: 脱クイズ度 4.0 / 実現性 3.0 (verdict: revise)
- **コンセプト**: リポジトリ全体で空白領域だった神経衰弱型を、コツコツムシのキャラ性 (「隠す→覚える→当てる」) をそのまま活かして導入する。8案中で最も脱クイズ度が高い。
- **置き換え対象**: コツコツムシ (kabuto) のシルエット当てクイズ (`silhouette`)。選択肢タップというクイズ解答形式そのもの。
- **遊び方**: 葉っぱ6枚 (3ペア) の下に虫の友だちを隠し、最初に全部を短時間見せてから伏せる。2枚ずつめくって同じ絵のペアを揃え、3ペア成立でクリア。ミス回数制限なし、めくり数が少ないと演出が豪華。文章ゼロ。
- **統合方法**: `_GAME_STARTERS` に `pair_flip` を追加し `kabuto.defaultGame` を差し替え。絵柄は既存 PEEK_QUIZ_POOL (16件) を流用でき新規イラスト調達不要。simon のセルCSSをクローン。勝利は `_resolveQuiz(true)` 一本 (敗北なし設計)。
- **レビュー懸念と対応方針**:
  - 実装検討の前提誤認: silhouette は「見せてから伏せる」タイマーを実は持たない (穴あき表示方式)。プレビュー→一斉非表示は新規実装として計上し直す (setTimeout ベース15〜20行、影響は軽微)。
  - 3歳児への負荷: 3ペア/1.5秒固定はワーキングメモリ要求が急上昇。**序盤2ペア/2.5秒→後半3ペア/1.5秒の可変化を必須要件にする。**
  - `_resolveQuiz` の音声 switch に `pair_flip` を追加し忘れると無音リリースになる (エラーにならず検知困難)。チェックリスト化。
  - 勝利コピー「せいかい！」を神経衰弱文脈 (「そろった！」) に合わせるか明示判断する。星演出ランク分岐は MVP から外し別チケット化。
- **工数**: S〜M (半日〜1日 + 上記修正分)。

### 1-2. てんびんいわの もん (おもさつりあわせゲート) — 推奨度: 中期検討

- **スコア**: 脱クイズ度 3.0 / 実現性 4.0 (verdict: revise)
- **コンセプト**: シーソー状の天秤に重り (どんぐり=1、くり=2) をドラッグで載せ、岩 (=3) と釣り合わせると門が開く。数の合成を体で覚える配置操作で、文章・選択肢なし。
- **置き換え対象**: 特定の虫の固定置換ではなく、後半ステージで同じクイズが繰り返される箇所に JSON の `minigame` 上書きで差し込む「第4の非クイズゲート」多様化枠。
- **統合方法**: 新 kind `balanceRock` + `balance_gate` をテーブル群に登録。傾きは CSS rotate 補間のみで物理エンジン不要。難度は JSON 側の `balanceTarget` / `balanceWeights` で制御。
- **レビュー懸念と対応方針**:
  - **最大の未決事項**: 「もん (通路封鎖ゲート)」を名乗るなら strengthRock / webGimmick と同じ重量級アーキテクチャ (stage gimmick 設定 + 共有 collision loop 編集) が必要で L 規模。徘徊クリーチャー版なら M。**着手前にオーナーがどちらか二択で決める。** これが本案を中期検討に置く理由。
  - パズル深度ゼロ問題: 重り2種各1個・目標3固定では正解が一択で「載せすぎ→戻される」状態に原理的に到達できない。**重り3種以上・複数解・おとりありへの再設計が前提条件。**
  - 「JSON書き換えだけで段階導入できる」は不正確 (初回に7〜8箇所のコード登録が必要)。企画文言を訂正。
  - 勝利は `_resolveQuiz(true)` ではなく独自 resolve + `_closeEncounter(true)` (虫バトル風の敗北演出が世界観に合わないため)。
  - 離脱/スキップ導線の設計、画像素材 (GPT Image 2) の AR 実測、CACHE_VERSION バンプを出荷条件に明記。
- **工数**: M (徘徊版) / L (真の門版)。

### 1-3. ゆれゆれゲート (タイミングくぐり) — 推奨度: 中期検討

- **スコア**: 脱クイズ度 3.5 / 実現性 3.0 (verdict: revise)
- **コンセプト**: おどけムシが揺らすツタの「あき」が開いた瞬間に大ボタンをタップして駆け抜ける、純タイミング勝負。文章ゼロ。
- **置き換え対象**: おどけムシ (odoke) の ○×クイズ (`truefalse`)。3種の中で最も文章読解負荷が高い瞬間。
- **統合方法**: flag 実装を雛形にテーブル4箇所へ `timing_gate` 登録。振り子位相は Date.now()/performance.now() 基準の絶対時刻で算出 (rAF 主体だとバックグラウンド間引きで見た目と判定がズレる既知の罠)。
- **レビュー懸念と対応方針**:
  - **年齢適合が本質的な論点**: ○×は「親が読み聞かせれば必ず解ける」救済経路を持つが、タイミング課題は運動スキルが未発達な3歳児に代替経路が無い。難易度を読解から反射神経に付け替えただけになるリスク。**「3回連続成功」のカウントをミスでリセットしない or 5連続失敗で自動成功 (pity) の救済をコードレベルで保証してから着手。**
  - `_resolveQuiz` も flag の HP 制 (`FLAG_HP_INITIAL=4`) も要件 (無制限リトライ・失敗状態なし) に合う既存パターンが無く、リトライ状態機械はゼロ設計。見積りを S→M (1〜2日) に上方修正済み。
  - iOS 実機でのタップ判定と視覚のズレは Playwright では検出不能。実機チューニングを独立サブタスクとして計画。開放区間の最小幅は3〜4歳の反応時間を実測ベースで決める。
  - imageStages/*.json 内の `minigame:'truefalse'` 明示指定の grep 洗い出しを着手前TODOに。
- **工数**: M (1〜2日 + 実機チューニング)。

### 1-4. ブンブンおとどけ わけっこ (ドラッグ仲間分け) — 推奨度: 中期検討

- **スコア**: 脱クイズ度 3.5 / 実現性 3.0 (verdict: revise)
- **コンセプト**: 散らばった5〜6個のアイテムを2つのカゴへドラッグで仕分ける。間違いカゴはぷるぷる拒否されて跳ね戻るだけで減点なし。カテゴリ認識という認知スキルは hachi の元コンセプトのまま、解答形式だけを操作に変換。
- **置き換え対象**: ブンブンムシ (hachi) の仲間はずれクイズ (`oddone`)。
- **統合方法**: water_bridge / donguri_bowl が確立したポインタドラッグ型パターンを踏襲。`sortTheme` フィールドでステージ毎にカテゴリ差し替え可能。
- **レビュー懸念と対応方針**:
  - **完了画面の再クイズ化バグ (レビューで検証済み)**: 計画の `_resolveQuiz(true)` 呼びは誤りで、汎用「せいかい！」画面が仕分けゲームの締めに出てしまう。water_bridge / donguri_bowl 同様、**専用完了画面 + `_closeEncounter(true)` 直呼びに設計変更 (必須)。**
  - 原案の touchstart/touchend 提案は memory の誤読。**Pointer Events + pointercancel を必須要件に格上げ。**
  - 複数アイテム同時ドラッグは repo 初の形。兄弟の指などによる第2タッチでの状態破壊を防ぐ**単一 pointerId 追跡を必須要件に追加。**
  - 詰み対策: カテゴリが分からない子が永久試行錯誤になる。同一アイテム4連続ミスで正解カゴをグロー表示するヒントを入れる。
  - 全 hachi 一括置換ではなく**ステージ単位のパイロット導入を正式なロールアウト計画とする。** 1エンカウンター5〜6ドラッグはテンポ低下の懸念もあるためアイテム数のステージ連動を検討。
- **工数**: 高M〜L寄り (2.5〜3.5日は楽観、完了画面新設とマルチタッチ対策込みで上振れ想定)。新規素材は AR 審査必須。

## 第2章 なぞなぞトレイン (nazonazo-tunnel/)

### 統括コメント: 「一律置換」ではなく「出題形式のバリエーション化」で段階導入する

なぞなぞトレインは既に sea (泡)、future (クレーン)、space (ネジ修理)、number (荷物ジャンプ) の4ステージが専用ミニゲームを持ち、**「ステージごとに出題形式が変わる」構造自体が確立済み**である。したがって本章の各案は「全ステージのなぞなぞを一斉に廃止する」ためのものではなく、**この既存バリエーションの5例目・6例目として、1ステージずつ追加していく**方針を統括として推奨する。

- 謎謎文はこのゲームの中核コンテンツ資産 (タイトルが「なぞなぞトレイン」) であり、全廃は自己否定になる。choice-card ステージを一定数残すこと自体が価値。
- Darius風格子 (総17ステージ/1プレイ10面、memory 参照) の分岐機構 (`resolveNextStage` 等) は全案で無改修と確認済みで、1ステージ単位の差し替えは構造リスクが低い。
- ただし cat / snow / ruins / sky はいずれも**隠し分岐ステージ (全プレイヤーが通るとは限らない)** であり、投資対効果の観点で「まず1本を隠し分岐で実験→手応えがあれば本線ステージへ横展開」という2段階を推奨する。

### 2-1. ねこの ひょっこりまど おぼえんぼクイズ (記憶フラッシュ) — 推奨度: すぐ着手候補 (条件付き)

- **スコア**: 脱クイズ度 3.5 / 実現性 3.5 (verdict: revise)。なぞなぞ側4案では総合最上位。
- **コンセプト**: 窓に2〜3匹のねこが約2秒ひょっこり顔を出して隠れ、「まんなかに いたのは だあれ?」と問う。repo 初の記憶系メカニクス。読解力ではなくワーキングメモリを使う。
- **置き換え対象**: cat ステージ (town→cat/fantasy 分岐の隠しステージ) の choice-card 5問。
- **統合方法**: `isCatMemoryStage()` + フラッシュ表示フェーズを挟むラッパー `renderCatMemoryGame()`。回答UIは既存 renderChoiceCards を流用でき、onPick 以降 (採点・乗客搭乗・進行) は無改修。number ステージと同じ「buildQList 早期 return + 実行時生成」パターン。
- **レビュー懸念と対応方針**:
  - 「見ていなかった」だけで不正解になる新種の不公平感。**再フラッシュ救済 (初回ミス時 or ?タップで1回だけ再表示) を必須要件に追加。level 0 は表示時間延長。**
  - 否定形の問い (「いなかったのは?」) は level 1-2 限定にし、level 0 は位置記憶のみ。
  - CAT bank は「猫の見た目バリエーション」ではないため MEMORY_CATS は実質新規素材。**まず絵文字フォールバックのみでプレイテストし、遊びとして成立してからイラスト投資する2段階ロードマップにする。**
  - `clearCatMemoryGame()` を正式成果物として showQuiz の既存クリーンアップ列 (6箇所) に追加。フラッシュ中の answerLocked 再ロックも明示項目化 (showQuiz が render 前に false リセットする実挙動のため)。
  - 演出は #quiz カード内の「窓風グラフィック」になる (背景世界と地続きではない)。期待値を事前にすり合わせる。
  - 隠し分岐への投資対効果は上記統括方針 (実験→横展開) で吸収。
- **工数**: 原案 S は過小、M で計画 (素材確認・年齢プレイテスト込み)。

### 2-2. いせきのポイントきりかえ にもつ仕分けレール — 推奨度: 中期検討

- **スコア**: 脱クイズ度 3.0 / 実現性 4.0 (verdict: revise)
- **コンセプト**: 貨車が1台ずつ来て、転轍機レバーで左右の線路に仕分ける。鉄道遊びの王道ギミックで電車世界観を強化。間違えても貨車が戻ってきて再挑戦。
- **置き換え対象**: ruins ステージ (future 分岐の隠しステージ) の choice-card 5問。
- **統合方法**: future/space と同型の拡張 (mechanic フラグ + showQuiz 分岐 + 専用データ SORT_SETS + 全貨車完了時に onPick 1回)。実現性は8案中最高クラス。
- **レビュー懸念と対応方針**:
  - 脱クイズ度は限定的 (既存4ミニゲームと同型の「お題差し替えタップ選択」の5例目という側面)。それ自体は悪くないが、新規性を過大評価しない。
  - **テーマ不整合が主課題**: 「たべもの/どうぶつ」例は遺跡世界観と無関係。**「ほんものの たからもの/にせもの」等、考古学文脈のカテゴリを先に確定し、既存 RUINS bank 8アイテムを可能な限り再利用する。** RUINS bank の温存/転用方針も同時決定。
  - トンネル分岐 (左右選択=失敗なし) と仕分けレバー (左右選択=正誤あり) が同じ操作文法を共有するルール混線リスク。レバーの形状・色を分岐UIと明確に差別化。
  - タイミング仕様の明文化: **貨車は静止してから操作を待つ (タイムアウトなし) を第一候補**とし、リアルタイム反応要素は入れない。
  - onPick の mode 別ハードコード (5箇所確認済み) への mode:"sort" 用文言追加、新規絵文字の quiz-art 登録有無の事前 grep、並走中の nazonazo 回帰テスト編集との衝突確認を着手前TODOに。
- **工数**: M+ (2〜4日、画像生成が絡めばさらに上振れ)。

### 2-3. そらの きてきエコー (汽笛リズムまねっこ) — 推奨度: 中期検討 (採点整合バグの修正が前提)

- **スコア**: 脱クイズ度 3.0 / 実現性 3.0 (verdict: revise)
- **コンセプト**: お手本の汽笛リズム (2〜4音、短/長) を聴いて・雲の視覚パターンで見て、汽笛ボタンで叩き返す。Web Audio 合成音のみで TTS 禁止ポリシーに抵触せず、外部音源不要。
- **置き換え対象**: sky ステージ (隠しステージ、rejoin: space) の choice-card 5問。
- **統合方法**: mechanic:"skyWhistle" + 実行時パターン生成。音は既存 tone()/AudioContext 基盤を流用。テンポ精度は問わず音数+長短一致のみの寛容判定。
- **レビュー懸念と対応方針**:
  - **レビューで発見された未検出バグ (最重要)**: 「失敗時に onPick を呼ばない」設計だと missInQ/stageMiss が一度も加算されず、**何度失敗しても常に3つ星+noMissボーナス確定**になり採点整合性が壊れる。UIクラス操作なしで加算だけ行う軽量パス `registerSkyWhistleMiss()` の新設を必須要件とする。
  - 乗客演出 (boardPassenger) 用の pe 情報の出所が未設計。**SKY bank を「表示しない乗客IDプール」として再利用**する案が最小工数で、既存コンテンツ資産も死蔵しない。SKY 配列の扱いを提案内で意思決定し TODO 化しない。
  - 5問とも「まねっこしてね」の同一指示になり、他ステージが持つ「5つの異なる謎」の新鮮さが失われる懸念 (脱クイズと引き換えの反復感)。

### 2-4. ゆきのまどふきクイズ (くもった車窓ワイプ) — 推奨度: 保留 (現設計のままでは不成立)

- **保留の理由 (致命的な設計不整合)**:
