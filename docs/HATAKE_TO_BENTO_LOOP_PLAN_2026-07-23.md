# 畑 → クッキング → お弁当 一気通貫プラン

作成日: 2026-07-23
対象: `hatake-nikki/` / `bento/kitchen.html` / `bento/index.html`
現状: プラン確定前の実装設計。畑の4区画構成・看板・水やり済み表示は batch:1431 までに先行実装。

## 1. 結論

「育てる → 収穫する → 料理する → お弁当に入れる」を一本の体験にする。初回は「とまと → プチトマト → お弁当」の縦切り1本で通し、次に「にんじん → にんじんいんげん」を追加する。

既存のクッキングとお弁当は「何度でも自由に作れる」遊びなので、畑在庫を参加条件にしない。畑の野菜がある時だけ「自分で育てた食材を使った」印と一気通貫ごほうびを追加する。

## 2. 体験フロー

1. 畑で種を選び、絵付き看板で区画を見分ける。
2. 水やりとお世話で育てる。
3. 収穫すると「はたけの やさい」が1こ増える。
4. 収穫後の非ブロッキング導線からクッキングへ移る。
5. 該当レシピに「はたけの やさい 1こ」の印を出す。
6. 料理を最後まで完了した時だけ、生野菜1こを「つくった おかず」1こへ変換する。
7. お弁当で同じおかずを置くと「つくった」印を出す。
8. お弁当の「できあがり」時に、実際に入っている畑由来のおかずを、種類ごとに最大1食分使い、一気通貫の完了演出を出す。

## 3. 作物・レシピ・お弁当の対応

| 導入順 | 畑の作物ID | 共通在庫ID | キッチン | お弁当名 | 理由 |
| --- | --- | --- | --- | --- | --- |
| 1 | `tomato` | `tomato` | recipe `tomato` | `プチトマト` | 既存の洗う・へたを取る・カップに入れる工程と直結し、最短で通せる |
| 2 | `ninjin` | `carrot` | recipe `ninjin_ingen` | `にんじんいんげん` | 「にんじんを育てて切り、いんげんと炒める」と工程が豊か |
| 3 | `ninjin` | `carrot` | recipe `kinpira` | `きんぴらごぼう` | ごぼうを畑作物に追加した後の発展に向く |

`hatake-nikki` の既存ID `ninjin` はセーブ互換のため変更しない。作物マスターへ `inventoryId: 'carrot'` を追加して吸収する。

## 4. UI方針

### 4.1 畑

- 畑は合計4区画。アイソメ上の2×2を「奥1・中央2・手前1」の左右対称なひし形に投影し、中央の緑のすき間をあぜ道として残す。
- 4区画は最初からすべて使用可とし、区画の段階解放は入れない。成長要素は土地の枚数ではなく、今後追加する種・料理・お弁当の対応で作る。
- 旧3区画セーブはindex 0〜2を保持し、index 3だけ空畑で補完する。保存キー `pono_hatake_state_v1` は変更しない。
- 看板は各 `.plot` の左角寄りに置き、種を植えた直後から収穫直前まで同じ絵を表示する。
- 空き区画と収穫直後は看板を隠す。
- 看板は `pointer-events: none`、装飾画像は `alt=""` / `aria-hidden="true"`。区画のひし形操作面は実button、または `role="button"` / `tabindex="0"` と Enter / Space / mouse click操作を備え、`aria-label` は「おく／ひだり／みぎ／てまえ＋作物名」で4区画を読み分けられるようにする。
- 水やり済み印は看板と共通の `.plot-marker` 内に置き、看板左上へ重ねて「この札の畑は水やり済み」と一体で読ませる。湿った暗色土と水色外周も併用し、印だけに依存しない。
- 虫は畝の右上、作物は中央を維持し、看板・水印のまとまりと役割を分ける。
- 収穫の宝箱を閉じた後、畑HUDに短時間だけ `クッキングへ` と `まだ おせわする` を出す。

### 4.2 クッキング

- 畑在庫がある対応レシピにだけ、小さな `はたけの やさい 1こ` 印を出す。
- 中途離脱、リロード、「もどる」では在庫を減らさない。
- レシピの完了コミット時だけ、生野菜を1こ減らし、調理済みの「1食分」を1こ増やす。
- 別タブで先に使われて在庫0になっても、料理の完了は止めない。その場合は通常の自由調理として完了する。

### 4.3 お弁当

- 調理済み在庫があるおかずカードに `つくった 1こ` 印を出す。
- 配置時は在庫を減らさない。置いてから取り消した時に「食材が戻る」特別処理を不要にする。
- `stageComplete()` 時に完成弁当内の名前を見て、対応する異なるおかずIDごとに最大1食分使う。同じおかずを複数置いても「1食分」の在庫は1つだけ使う。
- 在庫競合で消費できなくても、お弁当の完成は止めない。
- 完了処理は弁当ラウンド開始時に作った安定 `completionOpId` を使う。結果を `completionReceipt`（`consumedPreparedIds` を含む）として保持し、完成画面は減算後のlive在庫ではなくこのreceiptから「つくった」印と一気通貫演出を出す。

## 5. 推奨データ設計

作物の収穫やレシピ完了を「その場限りの解除」ではなく、共通在庫として保存する。新規キーは `pono_food_inventory_v1`。

```js
{
  version: 1,
  revision: 0,
  updatedAt: 0,
  raw: {
    tomato: { available: 0, earned: 0, used: 0 },
    carrot: { available: 0, earned: 0, used: 0 }
  },
  prepared: {
    petit_tomato: { available: 0, made: 0, packed: 0 },
    ninjin_ingen: { available: 0, made: 0, packed: 0 },
    kinpira: { available: 0, made: 0, packed: 0 }
  },
  appliedOps: {}
}
```

`appliedOps` は同じ操作IDの二重加算・二重消費を防ぐ。無限増大を避けるため、確定済みの新しい操作だけ128件程度にローテーションする。

### 5.1 複数タブでも失わない更新規約

`revision` と `appliedOps` だけでは、2タブが同時にread-modify-writeした時のlast-write-winsを防げない。すべての在庫mutationを次の共通経路へ集約する。

1. 収穫、料理run、お弁当roundの開始時に安定した `opId` を作り、各ゲームのセーブへ先に保存する。完了関数が二重に呼ばれても同じ `opId` を使う。
2. `navigator.locks.request('pono-food-inventory-v1', { mode: 'exclusive' })` の中で、在庫を再読込 → `appliedOps` 確認 → mutation → 保存する。
3. lock取得前に読んだ在庫値は判定に使わない。`revision` は監査と再試行検知に使い、排他制御の代わりにはしない。
4. Phase 1開始時に対象WebView / Safari / ChromiumのWeb Locks対応を実機確認する。未対応対象が1つでもある場合は、単純なlocalStorage leaseで代用せず、IndexedDB transactionを在庫の正本にして、非同期の書き出し／復元まで同Phaseで整備する。
5. `appliedOps` を削除できるのは、対応するproducer outboxがack済みで、復元データにも未処理opがないことを確認できる場合だけにする。

### 5.2 収穫と図鑑の消失防止

`hatake-nikki` の畝セーブと共通在庫は別localStorageキーなので、単純な2回書き込みだと中断時に収穫品を失う。畑セーブ側に `inventoryOutbox` を持つ。

1. 現在stateを直接破壊せず、「畝を空にする変更＋固有 `opId` 付き収穫操作」を含む次stateを作る。
2. `saveState(nextState)` は成功可否を返し、書き込みと読戻し確認に成功した時だけ画面stateを切り替える。失敗時は畝を空にせず、収穫演出も出さない。
3. outboxの同じ `opId` で、共通在庫への加算と図鑑count / firstDateをそれぞれ冪等適用する。図鑑側にも適用済みopを持たせるか、在庫の `earned` から導出する。
4. 在庫と図鑑の両方が保存成功した後だけoutboxをackして削除する。
5. 畑起動時に残ったoutboxを再実行する。

この方式なら、どの行間でアプリが閉じても、次回起動で回復し、二重には増えない。

## 8. tier方針

- App staging / app tier: 畑→クッキング→お弁当の連携を表示。
- LP staging の free / book tier: 畑由来印、導線、専用ごほうびを表示しない。既存のお弁当体験は変更しない。
- インポートデータに在庫があっても、それを理由に食材やレシピのtierロックを解除しない。
- MVPで `common/tier.js` のアンロック集合を変更しない。呼び出し側の `PonoTier.isApp()` で表示を制御する。

## 9. セーブ・同期・復元

- `pono_food_inventory_v1` をデータ書き出し／復元の対象にする。
- 旧端末からの復元でキーがない時は0在庫を作る。
- 不正な数値、未知ID、過大値は正規化する。在庫は0未満にしない。
- `common/data-export.js` の `CORE_PRESERVE_IF_ABSENT` とクラウド同期スナップショットの対象を見直す。
- Phase 2でbentoから新規common在庫モジュールを読む時点で、`native/content-manifest.json` と参照検証を更新する。畑本体をnativeへ入れる時は、別途 `hatake-nikki/` 一式も追加する。

## 10. テスト計画

### 畑／看板

- 667×375 / 844×390 / 1024×768 / 1366×768で4畝すべての4頂点が `green.png` のひし形面内。
- `#plot-area` のx座標／幅／中心／比率が、`background-size: contain` の `green.png` 表示面と一致する。
- 4畝の各中心が自身の操作面になり、中央のあぜ道と透明角が別の畝への誤操作を起こさない。
- 旧3区画セーブを読み込むと既存3区画を保持したまま4枠目だけ空で追加され、4枠目も植栽・水やり・収穫できる。
- キーボードだけで初回チュートリアル、種選択、植栽、水やりまで完走でき、マウスclickとtouchの合成clickが二重発火しない。
- 各区画を「おく／ひだり／みぎ／てまえ」の位置名つきで読み上げ、同じ作物を複数植えても区別できる。
- 390×844のタッチ縦画面は横向き案内が全面を覆う。
- にんじん／とまとで正しい看板が出る。空き区画と収穫後は消える。
- 看板とその左上の水やり済み印が一体で追従し、どちらも水やり長押し、虫フリック、収穫タップを遮らない。
- 看板＋水やり済み印のまとまりが、中央の作物・右上の虫・隣接畝を覆わない。

### 在庫

- 収穫で生野菜が1回だけ増える。
- outbox残存後の再起動で回復し、同じ `opId` は二重加算しない。
- 壊れた在庫JSONでクラッシュしない。
- 書き込み失敗時に収穫品を消失しない。
- 2 browser contextから同時収穫／同時調理しても加算・消費・`appliedOps`が失われない。

### クッキング／お弁当

- 料理途中の離脱・再読込みで生野菜が減らない。
- 料理完了で「生野菜 -1 / 調理済み +1」が1回だけ適用される。
- 同じ料理完了／`stageComplete()` を2回呼んでも、安定 `opId` により1回しか消費しない。
- お弁当で置いて取り消しても在庫は減らない。
- 完成時に実際に入った対応おかずがある時だけ、調理済みを1食分使う。
- 複数種類を入れた時は種類ごとに最大1食分を使い、receiptの `consumedPreparedIds` と完成演出が一致する。
- 在庫0でも従来の料理／お弁当遊びが完走する。
- free / bookでapp専用印やロック食材が漏れない。
- native同期後、`common/food-inventory.js` が404にならずbentoを完走できる。

## 12. 実装前の確定ゲート

以下はスキーマ変更を伴うため、Phase 1着手前にユーザーと最終確定する。

1. 畑由来食材を「消費する特別ルート」とし、従来の無限の自由遊びを残す。
2. 調理済み1こを「おかず1個」ではなく「お弁当に使える1食分」として扱う。
3. 最初の縦切りはとまとに限定する。
4. `pono_food_inventory_v1` と `inventoryOutbox` を新設する。
5. 完成弁当では、同じ種類を複数置いても1食分、異なる畑由来おかずは種類ごとに1食分を使う。
6. 全mutationをshared lock内で行い、Web Locks非対応対象があればIndexedDB transactionへ切り替える。
7. 収穫outboxは在庫だけでなく図鑑更新まで同じ `opId` で完了してからackする。
