## 1. 🌱 はじまりのはなし — なぜ アンケートを 取ることにしたのか

> **この章で 分かること**: なぜ アンケートを 取ろうと 思ったか、 その先の 判断に 使う 4 つの 問い。

いま ポノのあそびばは、 「本を出す前の 試し刷り」 のような時期に います。
これを **Early Access**（アーリーアクセス、 早期公開版のこと。 未完成だと わかった上で、
一部の 応援したい人に 先に 買ってもらう しくみ）と 呼びます。 値段は ¥980 です。

でも 本を 正式に 世に出す前に、 どうしても 知りたいことが 4 つ あります。

| 知りたいこと | どんな判断に つかう？ |
| :-- | :-- |
| 保護者さんは このアプリを 気に入って くれたか | 🚦 Early Access を **GO** するか **HOLD**（保留、 いったん止めて 直す）にするか |
| 5 つの ミニゲーム — quizland（クイズランド）/ maze（めいろ）/ oto（おと）/ bento（お弁当）/ puzzle（パズル） — の うち、 どれが 熱中されて、 どれが 冷めているか | 🎮 どのゲームを 太らせて、 どのゲームを 削るか |
| 絵本と アプリの つながりを、 保護者さんは 価値だと 感じているか | 📖 次の絵本に シールや 合言葉を もっと 仕込むか |
| 保護者さんは ¥980 を 高いと 感じたか 安いと 感じたか | 💰 Early Access の 値段を 変えるか |

これを 空想では なく、 実際の 保護者さんの こえで 決めたい。
それが このアンケートの 出発点です。

> **なぜ 空想では ダメなのか**
> プロダクトオーナーは 絵本作家です。 絵本作家の 直感は 大事ですが、
> 「未就学児の 保護者さんが お金を 払うか」 は 直感の 外の 話。
> だから 数字と 生のこえ、 両方を そばに 置いておきたいのです。

## 2. 🎁 こえの あつめかたのはなし — ★ボタンと 詳しい紙、 2 つの あつめ方

> **この章で 分かること**: 「★ 1 タップ だけの 人」 と 「9 問 答えたい 人」 を 分ける、 2 段構えの 玄関マット設計。

保護者さんに いきなり 9 個も 質問すると、 みんな 途中で 帰ってしまいます。
そこで こえの 集めかたを、 まるで お店の 玄関マットのように **2 段構え** に しました。

```
   ┌─────────────────────────────────────────┐
   │  ステップ 1  「ごかんそう」ボタン        │
   │  ─ ★を 1 つ 押すだけ ─                  │
   │                                         │
   │      ★ ★ ★ ★ ★  ←  タップ           │
   │                                         │
   │      ここで 8 割の 人は 帰る (でも OK)  │
   └────────────┬────────────────────────────┘
                │  「もっと 話したい」 と 思った人だけ
                ▼
   ┌─────────────────────────────────────────┐
   │  ステップ 2  survey.html                 │
   │  ─ 9 つの 質問に こたえる 紙 ─          │
   └─────────────────────────────────────────┘
```

### 🥄 玄関マット の役割（モーダル）

アプリの 中に 「ごかんそう」 という 小さな ボタンが 立っています。
これを 押すと **モーダル** （その場に すっと 浮き上がる 小窓のこと）が 開いて、
★を 1 つから 5 つまで 押せる 星が ならんでいます。

★を 押した 瞬間、 **その 星の 数だけが 匿名で 送信** されます。
保護者さんは 何も せずに、 その場で 閉じても 大丈夫。
これを **fire-and-forget**（撃ちっぱなし、 送ったら 返事を 待たない 送り方）と 呼びます。

### 📄 話したい人だけの 紙（survey.html）

モーダルの 下には 小さな 木の CTA（Call To Action、 「押してほしい ボタン」の 呼び名） が
「アンケートに こたえる →」 と 書いてあります。
これを 押すと、 別ページの **survey.html** に 移ります。
ここに 9 つの 質問が 待っています。

> **なぜ こう 分けたのか**
> 全員に 9 問 答えてもらう のは 無理。 でも 全員の ★ は ほしい。
> ★ だけなら 3 秒。 詳しく 話したい 人だけが 紙まで 進む。
> こうすると 数と 深さの 両方が 集まります。

## 3. ❓ 9 つの しつもんのはなし — 保護者さんに 何を きくのか

> **この章で 分かること**: survey.html に 並ぶ 9 問の 中身と、 特に Q7 で ¥980 を あえて 聞かない 理由。

survey.html に 並ぶ 9 つの 質問は、 順番も 数も 何度も 練り直して、 いまの 形に 落ち着きました。

| No | 質問の中身 | 必須？ | 形 |
| :-- | :-- | :-- | :-- |
| Q1 | お子さんの 年齢 | 必須 | ラジオボタン（3さい未満 / 3さい / 4さい / 5さい / 6さい / 7さい以上 / 複数のお子さん） |
| Q2 | 熱中度（どれくらい 集中して 遊んでいたか） | 必須 | ★5 段階 |
| Q3 | 遊びの様子を 見て 感じたこと | 必須 | チェックボックス、 9 つから 複数 選べる、 「良かった」 と 「気になった」 が 混ざっている |
| Q4 | つまづいた 場面 | 任意 | チェックボックス、 8 つから 複数 |
| Q5 | 一番 長く 遊んだ ゲーム | 任意 | チェックボックス、 5 ゲーム（quizland / maze / oto / bento / puzzle）から 複数 |
| Q6 | 絵本と アプリの つながり を 感じたか | 任意 | 4 段階 ラジオ |
| Q7 | 絵本 ¥1,500 と くらべて、 この アプリの 価値は | 任意 | 5 段階 ラジオ（¥980 を 直接 聞かない） |
| Q8 | 自由記述（500 文字まで） | 任意 | テキスト |
| Q9 | Early Access 招待メール を 受け取りますか | 任意 | メールアドレス |

### 🤫 Q7 で「¥980」を あえて 聞かない 理由

「¥980 は 高い？ 安い？」 と 聞いてしまうと、 保護者さんは
「聞かれた 値段が 正解 なんだろう」 と 思って **アンカリング**（最初に 見た 数字に
判断が 引きずられる 現象）してしまいます。

だから 別の モノサシで 聞く。 絵本 ¥1,500 は 保護者さんが すでに 触れている 値段。
それと 比べたときに どう 感じるか を 聞くと、 素の 価格感度が わかります。

### 🎭 Q3 に ポジ と ネガを 混ぜている 理由

ポジ（良かった） だけを 並べると、 「なんとなく 全部 チェック」 が 起きます。
ネガ（気になった） だけを 並べると、 「全部 大丈夫でした」 で 素通り されます。
両方を 混ぜると、 保護者さんは 立ち止まって 一つずつ 選び直します。 そこに 本音が 出ます。

### 🔒 匿名性の おやくそく

Q9 の メールアドレス を 除いて、 個人を 特定する 情報は 一切 聞いていません。
Q9 で メールを 書いた 場合も、 それは 「Early Access 招待 の 連絡先」 に しか 使いません。
保護者さんの 端末 に 残る **anonSid**（次章）も、 名前や 顔ではなく
「同じブラウザ」 の しるし だけです。

## 4. 🗄️ データの すみかのはなし — 16 の 引き出しと 3 つの 部屋

> **この章で 分かること**: こえが しまわれる Google Sheets の 部屋割りと、 16 個の 引き出しの 中身。

集めた こえは、 **Google Sheets**（Google が くれる 表計算ノート、 Excel の親戚）
1 冊に すべて 集まります。 この 一冊が、 プロダクトの 心臓です。

### 🚪 3 つの 部屋（タブ）

ノートの 中には、 タブ（表紙めくり）が 3 つ 自動で 作られます。

```
  ┌────────────┐  ┌────────────┐  ┌────────────┐
  │  staging   │  │ production │  │  unknown   │
  │  試し部屋  │  │  本番部屋  │  │ 忘れ物入れ │
  └────────────┘  └────────────┘  └────────────┘
```

- **staging**（ステージング、 開発中の お試し版のこと） — 開発中の URL から 送られた 声
- **production**（プロダクション、 本番版のこと） — 正式リリースの URL から 送られた 声
- **unknown**（アンノウン、 わからない） — どちらか 判別できなかった 送り主。 捨てないで ここに 入れる

なぜ **unknown を 捨てないのか**。 「わからないから 捨てる」 と、 バグで 環境情報が 落ちた
瞬間に データが 全部 消えます。 一度 「バケツ」 に 入れて、 あとで 目視できるように。
これが プロダクトオーナー流の 保険です。 送り主の hostname 判別が 失敗した ときの、
最後の 受け皿 でも あります（判別 の しくみは §5 で 詳しく）。

### 📐 16 の 引き出し（列）

1 行が 1 人の こえ、 縦に 16 個の 引き出し（列）が 並びます。
中身を 「👶 保護者さんが 答える もの」 と 「🤖 しくみが 勝手に 書く もの」 に 分けて 見ると
読みやすい です。

#### 🤖 しくみが 勝手に 書く 引き出し

| # | 引き出しの名前 | 中身 |
| :--: | :-- | :-- |
| 1 | timestamp | 送られた 時刻 |
| 2 | environment | staging / production / unknown |
| 3 | anonSid | **匿名 ID**（誰か 名前は わからないけど、 「同じブラウザ」 だとは わかる 識別子） |
| 5 | gameId | どの ゲームから 送ったか |
| 6 | swVersion | **サービスワーカー**（アプリを 端末に キャッシュする 縁の下の仕組み）の バージョン |
| 7 | playDurationSec | 遊んだ 時間 (秒) |

#### 👶 保護者さんが 答える 引き出し

| # | 引き出しの名前 | 対応 |
| :--: | :-- | :-- |
| 4 | starScore | ★の数 (1〜5)、 モーダル から |
| 8 | age | Q1 |
| 9 | engagement | Q2 |
| 10 | positiveNotes | Q3 |
| 11 | stumbles | Q4 |
| 12 | favoriteGames | Q5 |
| 13 | bookConnection | Q6 |
| 14 | pricePerception | Q7 |
| 15 | freeText | Q8 |
| 16 | email | Q9 |

### 🔁 同じ人が 何度も 押さないための 2 段防御

「ごかんそう」 ボタンを 連打されると、 データが ゴミで 溢れます。 それを 防ぐために
2 段構え の 防御を 入れています。

1. **同日重複送信 flag**（localStorage 内） — 送信した 日を **localStorage**（保護者さんの ブラウザに ちょっとだけ 記憶できる 小さな 引き出し）に 印を つけ、 同じ日は もう 送れない
2. **anonSid**（列 3） — 「同じブラウザ」 の しるし。 後で 集計時に 重複行を 束ねる ための 目印

localStorage の 印 は 「その場で 弾く 見張り」、 anonSid は 「あとで 集計する ときの 名札」。
役割が 違うので 2 つ ある、 と 覚えてください。

## 5. 📮 ゆうびんやさんのはなし — Apps Script という 郵便局員

> **この章で 分かること**: ブラウザ と Google Sheets の 間に 立つ 郵便局員 (Apps Script) が、 何を してくれるのか。

保護者さんの ブラウザから Google Sheets に 直接 手紙を 届けるのは、 実は とても 難しい。
なぜなら 途中に セキュリティの 壁が いくつも 立っているからです。

そこで 間に **郵便局員** を 雇いました。 それが **Apps Script**（アップス・スクリプト、
Google が 用意している 「小さな サーバの中の人」 サービス）です。

```
   保護者さんの ブラウザ
        │
        │  「これ 届けて」 (POST)
        ▼
   ┌───────────────────┐
   │   Apps Script     │  ←  郵便局員
   │  お手紙 受取係     │
   └────────┬──────────┘
            │  1) 環境（staging / production / unknown）を 判別
            │  2) 引き出し 16 個に 中身を 仕分け
            │  3) 初回なら 列名も 自動で 書き込む
            ▼
     Google Sheets （心臓）
```

### 🛡️ 郵便局員に 持たせている 5 つの 安全策

保護者さんの こえは 大事に 扱いますが、 悪意の 手紙は 弾かないと いけません。
郵便局員には、 **3 つの 見張り** と **2 つの 心づかい** を 持たせて あります。

**3 つの 見張り（悪意手紙を 弾く）**

1. **サイズ制限** — あまりに 大きな 手紙は 受け取らない（payload 制限）
2. **文字数制限** — 500 文字を 超える 自由記述は 切り詰める
3. **排他ロック**（LockService、 一度に ひとりだけ 書き込ませる 仕組み）— 同じ 一行に 2 人が 同時に 書き込んで 事故が 起きないよう、 順番待ちを させる

**2 つの 心づかい（大事な 声を こぼさない）**

4. **環境不明値を 捨てない** — 判別 に 失敗しても 捨てず、 unknown 部屋 に 入れる
5. **ヘッダー自動投入** — 初回の 手紙が 届いたとき、 引き出しの ラベル（列名） も 自動で 書き込む。 手作業で 列名を 入れる 必要は ない

### 📬 URL は 1 本、 判別は クライアント側

送り先の URL（Apps Script の 玄関） は 1 本だけ。
staging から 送られたのか production から 送られたのかは、
保護者さんの ブラウザが 自分の hostname（今 見ている アドレスの 名前）を 見て、
自分で 判定して、 荷札に 書いてから 送ります。

hostname を 読み損ねた 手紙 は、 郵便局員が 「わからない」 と 判定して
§4 の unknown 部屋 に 入れます。 これが 「なぜ unknown 部屋が あるか」 の 答えです。

> **なぜ 郵便局員を 挟むのか**
> Google Sheets の 玄関に 直接 手紙を 投げると、 誰が 送ったか 追いにくく、
> セキュリティ制御も できません。 郵便局員を 一人 挟むだけで、 見張りが 立ち、
> 荷札の チェックが できるようになります。 これが 一番 素朴で 一番 堅い やり方です。

## 6. 🎨 みための おんどのはなし — なぜ Google フォームを 使わなかったのか

> **この章で 分かること**: モーダルと survey.html の 見た目を、 なぜ 手作りしたか。

アンケートを 集めるだけなら、 Google フォームを 使えば 数分で できます。
でも 使いませんでした。 理由は 「絵本の 温度」 が 保てないからです。

### 🧡 モーダル（★ボタンの 小窓）の みため

| 要素 | どう したか |
| :-- | :-- |
| 背景色 | クリーム #FFF8F0 （絵本の 紙の色） |
| アクセント色 | オレンジ #F2915A （Pono の 帽子の色） |
| フォント | Zen Maru Gothic （まる い ひらがな） |
| 角丸 | 22〜28px （角ばらせない、 やわらかい） |
| CTA ボタン | 木製 #FFD84D （踏むと 音が しそうな 黄色） |

### 📖 survey.html の みため

同じ ブランドトーンで 統一。 ヘッダーには brand_logo.png（横長の ロゴと キャラ）が
そっと 顔を 出して、 「あぁ Pono の 世界の 続きだな」 と 保護者さんが 感じられるように しました。

> **なぜ そこまで こだわるのか**
> Google フォームの 見た目は、 銀行の 書類のように 冷たい。
> Pono の 絵本を 買った 保護者さんに 「銀行の 書類」 を 差し出したら、
> それは もう 別の 会社の アンケートです。 温度が 変わると、 答える 気持ちも 変わる。

## 7. 🔍 かずを よむのはなし — 集めたこえを 数字に する

> **この章で 分かること**: Sheets の 中に 建てる 6 つの 「読むための 部屋」 と、 週次要約を 書く AI の 選び方。

集めた こえを ながめる だけでは、 判断が できません。
「数える」 「並べる」 「気になる 一行を 拾い上げる」 — この 3 つの 動作を、
Google Sheets の 中で ぜんぶ 完結させます。 これを **Approach A（Sheets ネイティブ）**
と 呼んで、 3 つの 案（A / B / C）から 選びました。

### 🥇 3 つの 案 と なぜ A に したか

| 案 | ざっくり | 追加インフラ | 実装時間 | 月額 | 結論 |
| :-: | :-- | :-- | :-- | :-- | :-- |
| A | Sheets の 中で 完結 | ゼロ | 13 時間 | $0.09 or 無料 | 🔒 **今すぐ 採用（合意ずみ）** |
| B | Looker Studio （見た目 きれい ダッシュボード） | ちょっと | +8-10 時間 | 無料 | Phase 3 で 追加 |
| C | 自前 admin ダッシュボード | 大きい | 数十時間 | サーバ代 | ❌ 今の 規模には 大きすぎる |

**Looker Studio**（Google が くれる ダッシュボード作成ツール）は 綺麗ですが、
今は データが 少ないので 使っても 意味が 薄い。 データが 溜まった Phase 3 に 回します。

### 🏠 Sheets の 中に 建てる 6 つの 部屋（タブ）

これは 分析用に、 データ収集用の 3 部屋（§4）とは 別に 建てる 「読むための 部屋」 です。

| 部屋 | 何が 見える |
| :-- | :-- |
| Dashboard（ダッシュボード、 一望する 部屋） | KPI（Key Performance Indicator、 「大事な 数字」）を 12 枚 + グラフ 4 種 |
| Game_DeepDive（ゲーム深堀り） | 5 ゲーム × ネットスコア（favorite − stumble、 好きの数 から つまづきの数 を 引いたもの） |
| Trend_Weekly（週次トレンド） | 週ごとの 3 指標 折れ線グラフ |
| Weekly_Summary（週次サマリ） | Apps Script と AI が 週次要約を 自動で 書き込む 部屋 |
| Decision_Log（決定ログ、 監査ログ） | プロダクトオーナー **本人** が 週次の 判断を 3 行以内 で 書く |
| _Helpers（お手伝い） | 多選択の 列を 展開する 裏方タブ |

> **Decision_Log は AI に 代筆させない**
> ここは 「オーナー が いつ 何を 決めたか」 を 追う 監査ログ です。
> Claude や Codex に 書かせると、 判断者が ぼやけて 意味を なくします。
> 30 分の 儀式（次章）の 最後 3 分で、 必ず 手で 書くこと。

### 🤖 AI 要約は Gemini 2.5 Flash 🔒 確定 (2026-07-04)

Weekly_Summary の 中身を 書いてくれる AI は **Gemini 2.5 Flash** に 決めました。

| 候補 | 品質 | 月額 | 準備 の 手間 | 採否 |
| :-- | :-- | :-- | :-- | :-- |
| **Gemini 2.5 Flash** | そこそこ | **無料** | Google の 中の子 なので Apps Script との 相性が 一番 いい | ✅ 採用 |
| Claude Haiku 4.5 | 高い | $0.09 | Anthropic API キー が 必要 | 参考 |

**なぜ Gemini か**:

1. **無料** — 月に 数百件の 要約なら 無料枠の 中に すっぽり 収まります
2. **準備が いちばん 少ない** — Apps Script は もともと Google 製品 の 内側に います。 Gemini の 呼び出しは、 別会社の API キーを 発行して どこかに 貼り付ける 手間が いりません
3. **絵本作家さんの 手間が ゼロ** — アカウントを もう 1 つ 増やしたり、 クレジットカードを 登録したりする 作業が 発生しません

**Claude Haiku を 選ばなかった 理由**: 品質は Haiku の 方が やや 上ですが、 週次要約 という 用途では 「そこそこ」 で 十分。 品質差の 対価として、 API キー 管理と 月 $0.09 の 支出を 引き受ける 価値は 現段階では 見送り。 将来 「もっと 賢い 要約が 欲しい」 と 感じたら Claude に 差し替えできる 構造 に します（プロンプトは 共通、 呼び出し先だけ 差し替え可）。

## 8. 🚦 GO か HOLD かのはなし — 毎週月曜 30 分の 儀式

> **この章で 分かること**: 毎週 30 分の ルーティン と、 GO を 出す 5 つの 数字（動かさない）。

数字を 読んで、 判断を くだす。 これを 毎週月曜 の 朝、 30 分の 儀式 に します。

### 🕘 30 分の 内訳

```
   0 分 ─┬──────────────────────────────────
         │  0–2 分 :  Dashboard で 今週の 直観
   2 分 ─┤
         │  2–7 分 :  Trend_Weekly で 3 週の 傾向
   7 分 ─┤
         │  7–12 分 : Weekly_Summary の AI 要約 を 読む
  12 分 ─┤
         │ 12–17 分 : ★1-2 の 生の 一行 を 読む
  17 分 ─┤
         │ 17–27 分 : Game_DeepDive で 「今月 伸ばす / 削る」 判断
  27 分 ─┤
         │ 27–30 分 : Decision_Log に 3 行 以内 で 書く
  30 分 ─┴──────────────────────────────────
```

### 🎯 🔒 GO の 5 条件（4 週連続で 全部 満たしたら Launch — 合意ずみ 動かさない）

Early Access ¥980 を GO する 条件を、 **合意ずみで 動かさない** と 決めています。
なぜ 動かさないか。 数字が 悪くなった ときに 「今月は 特別」 と 動かすと、
そもそも 閾値の 意味が なくなるからです。

| 名前 | 中身 | 閾値 |
| :-: | :-- | :-- |
| K1 | 星の 平均 | ≥ 4.0 |
| K2 | ★4-5 の 比率 | ≥ 60% |
| K3 | ★1-2 の 比率 | ≤ 15% |
| K4 | 熱中度（Q2） の 平均 | ≥ 3.8 |
| N | 回答数 | ≥ 80 |

「4 週連続」 が ミソです。 一週間 だけ 良くても、 それは たまたま かもしれない。
4 週 続けば、 それは 実力です。

### ⚖️ HOLD は NO-GO では ない

もし 条件を 満たさなかった とき、 これは **HOLD**（ホールド、 保留）です。
「もう ダメだ、 やめよう」 の NO-GO では ありません。

HOLD の あいだに やることは 決まっています。

1. どの 数字が ダメ だったか 特定 する
2. 一番 効く 改修を 選ぶ
3. 直す
4. また 4 週 測る

つまり HOLD → 直す → 再測定 の ループです。

```
   ┌─────────────┐   条件 満たさず   ┌────────────────┐
   │  4 週 測る   │ ─────────────▶ │  HOLD (保留)   │
   └──────▲──────┘                  └────────┬───────┘
          │                                   │
          │   直したら                        │  原因特定
          │                                   ▼
          │                          ┌────────────────┐
          └───────────────────────── │  改修を 実装   │
                                     └────────────────┘
```

### 📬 週次デジェスト の 届き方

毎週月曜 9:00 JST（日本時間） に、 2 つの 場所に 同じ 内容が 届きます。

- Weekly_Summary タブ に 自動追記
- Gmail 自宅アドレス に メール送信

「Sheets を 開くのを 忘れても、 メールが 追いかけてくる」。
30 分の 儀式を すっぽかさない ための、 やさしい 見張り番です。

## 9. 🛤️ これからの みちのりのはなし — Phase 1, 2, 3 と ここまでの 足あと

> **この章で 分かること**: 3 つの 季節 (Phase) の 作業内訳 と、 これまでに 送信ずみ の 3 つの batch。

このアンケートの しくみは、 一気に 建てる ものでは ありません。
畑を 耕して、 種を まいて、 実を 見て、 増やす。 3 つの 季節に 分けます。

### 🌱 Phase 1 ── 今週、 13 時間の 作業

「畑を 耕す」 の 季節。 骨組み を 全部 建てる 段階。

- Sheets Dashboard の 部屋 6 個 を 建てる
- Apps Script の 週次バッチ（毎週月曜 朝 動く スケジュール実行） を 仕込む
- Gmail 送信 を つなぐ
- GO/NO-GO 判定セル を 作る

### 🌿 Phase 2 ── 回答が 30 件 貯まったら、 3 時間

「種の 育ちを 見る」 の 季節。 実データが 少し 出て きた ときに、
AI 要約が ちゃんと 意味の ある 文章を 書けているか、 閾値は 現実に あっているか を 見直します。

> **§8 の 「動かさない」 と 矛盾しないか**
> Phase 2 の 「閾値 見直し」 は、 GO を 出す 直前 に 動かす こと では ありません。
> **Phase 1 で 仮置き した 値が 明らかに 現実離れ していないか** を、
> Launch 前に 一度だけ 検算する ゲート です。
> Launch 後、 4 週 測っている 途中で 動かすのは 依然 禁止 です。

### 🌳 Phase 3 ── Early Access launch 後、 N≥100/月 に なったら、 8-10 時間

「実を 増やす」 の 季節。 Looker Studio で 4 ページの きれいな ダッシュボードを 建てて、
将来の 投資家に 見せられる URL を 用意します。

### 🐾 ここまでの 足あと（batch 履歴）

このドキュメントを 書く 直前 まで に、 送信済みの 修正バッチ が 3 つ あります。
（それ以前 の batch は 別ジャンル の 修正 が 混ざるので、 ここでは 「survey 系」 に 絞って 3 つ 記載）

| batch | 何を したか | サービスワーカー バージョン |
| :--: | :-- | :-- |
| **batch:936** | 星評価の 隠し POST + survey.html を 新設 | v1950 → v1951 |
| **batch:937** | モーダル表示バグ 修正（iOS Safari で モーダル が 透明のまま 固まる 問題） | v1951 → v1952 |
| **batch:938** | swVersion（サービスワーカー バージョン） が 空文字 に なる 問題 修正 | v1952 → v1953 |

**サービスワーカー** は アプリを 端末に キャッシュする 縁の下の仕組み で、
コードを 変えたら バージョンを 上げないと 保護者さんの 端末に 新しいものが 届きません。
だから バッチごとに バージョンが 上がっています。

## 10. 🌫️ まだ みぬ もんだいのはなし — 先送りに した 3 つの こと

> **この章で 分かること**: 「今の 規模では まだ 起きない」 と 判断して、 わざと 後回し にした 3 つの 課題。

正直に 書きます。 いま **わざと 先送りにしている** 課題が 3 つ あります。
理由は 「今の 規模では まだ 起きない」 「別バッチで やる ほうが 綺麗」 だからです。

### 🚧 課題 1 — Apps Script レート制限 と CAPTCHA（DoS 対策）

**DoS**（Denial of Service、 悪意ある 大量アクセス で サーバを 潰す 攻撃） を
防ぐために、 郵便局員（Apps Script）に 「1 分に N 通まで」 の 上限を つける 話。
今は アプリの 利用者が 少ないので、 まだ 起きません。
起きる 頃には 別バッチで やります。

### 🚧 課題 2 — CSP ヘッダに connect-src script.google.com を 追加

**CSP**（Content Security Policy、 「どの アドレスに 通信して いいか」 を ブラウザに 教える 決まり） の 話。
Apps Script に 通信できる 相手として 明示的に 許可を 書き足す 必要が あります。
worker.js（Cloudflare の 番人）の 改修 が 要るので、 別バッチで やります。

### 🚧 課題 3 — ★1-3 auto-close の 幼児 UX 検討

低評価（★1-3）の ときに モーダルを 自動で 閉じる 挙動が、 幼児が 触ると
どう 見えるか。 実機テストの 結果を 待って、 変更するか 決めます。
これは 数字では なく、 実際に 3 歳児が 触っている 動画を 見ないと わからない 話。
