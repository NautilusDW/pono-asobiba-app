# ポノあそびば 新ジャンルゲーム企画 最終提案レポート

**本書は提案書であり、この時点で実装は一切行わない。** 以下は7ジャンル×3案＝21件のアイデア、各案の実装検討、および別エージェントによるクロスレビュー(全21件が verdict: revise)を統括した最終判断である。

> 生成プロセス: リポジトリ調査(sonnet, 3並列) → ジャンル切り口設計(fable) → アイデア発想(fable, 切り口ごと並列) → 実装フィージビリティ検討(sonnet) → クロスレビュー(sonnet, 発案者と別エージェント) → 最終統合(fable)。計54エージェント。

## 1. 課題認識とジャンル切り口設計の要約

### 課題認識

現行ラインナップは quizland / nazonazo-tunnel / quiz-sound など「出題→回答→正誤判定」の一問一答構造に強く偏っている。この構造は (1) 文字・知識への依存が高く低年齢層の下限を切り捨てやすい、(2) 遊びのリズムが「止まって考える」に固定される、(3) コンテンツ追加が問題量産に律速される、という3つの限界を持つ。

### ジャンル切り口設計

偏りを構造から崩すため、以下7つのレンズで各3案を発想させた。

| レンズ | 狙い |
| --- | --- |
| アクション/反射神経系 | 「考える」ではなく「体が反応する」 |
| リズム/音楽系 | 正誤のない身体的コール&レスポンス |
| 物理・構築チャレンジ系 | 答え合わせの相手が物理法則のみ |
| 育成・お世話シミュレーション系 | 時間と継続を軸にした関係構築 |
| 探索・観察サンドボックス系 | 世界に働きかけて眺める双方向性 |
| 創作・表現系 | 子供のアウトプットが作品になる |
| 戦略・かけひき系 | 相手との相対で最善手が変わる |

## 2. 評価の観点と横断的な技術所見

優先順位は「差別化度(diversity)×実装フィージビリティ(feasibility)」のレビュースコアを基軸に、レビューが検出した事実誤認の重さを加味した。**なお全21件が verdict: revise であり、無条件で着手可能な案は1つもない。** どの案も着手前に本書記載の「対応方針」の消化が前提となる。

またレビュー横断で、複数アイデアの前提を崩す**共通の事実誤認・共通制約**が確認された。以降の全案に適用される。

1. **`common/drag-helper.js` は存在しない。** 複数案が「共用」を前提にしていたが未実装。ドラッグ系ゲームの1本目がこれを新規に書くことになる(iOS pointercancel 対策 = touchstart/touchend/touchcancel 追跡必須、MEMORY 既知バグ)。
2. **`common/parallax.js` (ParallaxScene) は本番未投入。** 読み込んでいるのは tools/bg-editor.html のみ。nazonazo-tunnel の背景は独自実装。「実運用済み」を前提にした案は要修正。
3. **`assets/images/ocean/` は各生物1枚絵のみ**(`<Name>_normal_1.png`)で surprised/happy 等の表情差分は無い。「表情差分流用」を謳う案はすべて GPT Image 2 での新規生成が必要。
4. **`assets/images/characters/racoon/` は未整理の生レイヤー4枚**(出所不明・AR不揃い)。あらいぐまを対戦相手にする案は素材作り直しが前提。
5. **`aquarium/dynamic-creatures.js` は PIXI.js 依存**。Canvas 2D 系ゲームへは「設計思想の流用」に留まりコード移植は不可。
6. 共通遵守事項: sw.js CACHE_VERSION バンプ(新規ディレクトリ追加時 MUST)、画像は GPT Image 2 固定+AR実測遵守、キャラ肉声禁止(第三者ナレのみ、一人称テキスト吹き出しも回避)、tests/ 配下の回帰テスト作成をどの案も工数計上漏れしている。
7. **リズム系全案が oto/ (16,561行、rhythm-layout・taiko音色・ノーツスケジューラ実装済み) を調査対象から漏らしていた。** リズム枠は新規ゲームではなく oto 拡張として再定義すべきというのが本書の判断。

## 3. 統合・整理の結果

内容が重複する案は以下のとおり統合した。

| 統合後 | 統合元 | 理由 |
| --- | --- | --- |
| リズム枠は「oto 幼児向けモード」として再定義 | まねっこポンポン + もりのオルゴールばこ | 両案とも oto の既存リズム機構と機能重複。新規ディレクトリを起こす合理性が薄い |
| パレード案(1本化) | おんがくパレード + わくわくパレード | 「パートナー動物が行進+音楽」の核が同一。かつ両案とも横歩きスプライトが8種中ほぼ皆無という同じ致命的素材欠落を抱える |
| お世話枠は「はたけにっき」1本に絞る | もりのおへやケア / そだてよすいぞくかん は保留 | 3案ともたまごっち型の同型構造。動物ケア2案は表情差分素材ゼロ+ネグレクト演出リスク+tier未決というブロッカーが重く、植物なら「しおれても罪悪感が軽い」設計が最も成立しやすい |
| 探索枠のタップ反応系は aquarium 拡張候補へ | つんつんの森 | コアループが aquarium 実装済み機能と同一と実測確認されたため独立タイトル化を取り下げ |

## 4. 推奨アイデア詳細

### 4.1 すぐ着手推奨(Tier A)

#### A1. どんぐりわけっこ対決(3つのカゴの資源配分)— 戦略・かけひき系

- **レビュー**: revise / diversity 4 / feasibility 3(21件中、差別化と実現性の両立が最良)
- **コンセプト概要**: ライト版 Colonel Blotto。どんぐり5個を山・川・森の3カゴにドラッグ配分し、あらいぐまNPCと同時公開でカゴごとの多数勝負。相手の「川をじっと見ている」等の非言語ヒントを読んで裏をかく。数字も文字も不要、3ラウンド勝負→宝箱。
- **流用する既存資産**: `common/haptics.js`(PATTERNS追加)、`common/treasure.js`(showTreasure)、`common/achievements.js` / `common/stickers.js`(実績・シール)、`assets/audio/battle/victory_fanfare.mp3`、`assets/images/yard/`・`assets/images/tree_stone_flower/`(背景装飾)、`maze/_sim.js` の「生成→監査→リトライ」設計思想(コード移植ではない)。※レビューによる実ファイル確認で、これら common 系のAPI形状記載は正確と裏付け済み。
- **新規開発が必要な部分**: ゲーム本体JS 600〜800行(ドラッグ&ドロップ、勝敗判定、NPCヒント演出、旗UI)、あらいぐまの表情差分3〜5点+カゴ/旗/どんぐり素材(GPT Image 2)、回帰テスト。既存流用:新規 ≒ 15:85。
- **クイズ系との差別化**: 出題→回答の往復が存在せず、子供が先に「問い(配分)」を作る側に回る。正解は相手との相対で決まり、毎回答えが変わる。
- **懸念点と対応方針**:
  - 21通りしかない配分空間+単一ヒントで戦略が浅くなる懸念 → プレイテストで再プレイ意欲を検証する前提を明記。NPC配分は「事前生成固定パターン+シャッフル」に単純化してスコープを絞る。
  - racoon素材が生レイヤー4枚のみ → GPT Image 2 で完全新規生成を前提に計画。
  - 連敗時の救済設計なし → 難易度緩和 or 参加賞シールを仕様に追加。
  - 3カゴ横並びのARリスク → カゴ3つを1枚の背景アートとして一体生成し構造的に回避。
  - テスト工数未計上 → M+ に補正して見積もり直し。
- **推奨度**: **すぐ着手**(上記の設計確定を先行タスクとして)。

#### A2. ぱくぱくキャッチ!(落ちもの受け止め)— アクション/反射神経系

- **レビュー**: revise / diversity 3 / feasibility 3
- **コンセプト概要**: 落下する食べ物を1軸左右ドラッグのカゴで受け止める60秒タイムアタック。イガグリ・石は「受けてはダメ」の go/no-go 判断。失敗はコミカル演出のみ。アクション枠の最小パイロット。
- **流用する既存資産**: `bubble/index.html` の choosePentaFreq / playMarimbaNote(コンボ音、要改修)・addParticle / addConfetti(コピー移植)、`common/highscore.js`(saveHighScore)、`common/haptics.js`、`common/achievements.js` / `common/stickers.js`、`assets/images/characters/pono/pono_001.png` 等。
- **新規開発が必要な部分**: 新規ディレクトリ一式、落下スポーンテーブル、**1軸ドラッグ+iOSタッチ追跡(前例なし・新規)**、落下物素材。**ocean/ に「どんぐり・おにぎり・イガグリ・石」は実在しないことが確認済み**で、落下物5種中4種以上は GPT Image 2 新規生成が必須。カゴ持ちポノも新規。
- **クイズ系との差別化**: 知識ゼロ、見た目(トゲトゲ)による瞬時の抑制判断のみ。プレイ中に文字が1文字も出ない。
- **懸念点と対応方針**:
  - 素材流用前提が誤りだった → scope を S から M 寄り(2〜3人日)に補正済みの前提で計画。
  - bubble との手触り酷似リスク → 音色・演出を bubble と意図的に変える工夫を企画に追記。
  - 難易度構造(単一加速か選択式か)未決 → 着手前に単一自動加速カーブに決め打ち(highscore.js をそのまま使える)。
  - ここで書く 1軸ドラッグ実装を、将来の `common/drag-helper.js` の種にする(ただし共通化自体は別チケット)。
- **推奨度**: **すぐ着手**(A1 と並行可。ドラッグ基盤の先行投資を兼ねる)。

### 4.2 中期検討(Tier B)

#### B1. ひょっこりハイタッチ(ポジティブもぐらたたき)— アクション/反射神経系

- **レビュー**: revise / diversity 4 / feasibility 2.5
- **コンセプト概要**: 隠れ場所からひょっこり出るパートナー動物を素早くタップしてハイタッチ。zzz で眠る子はタッチしない go/no-go。据え置き画面での瞬発タップはラインナップに現状ゼロ。
- **流用する既存資産**: `assets/images/characters/headgehog/` の8枚(front/surprised/smilewavinghands 等)、`assets/images/puzzle/partners/partner_*.webp`(8種・ただし静止1枚絵)、`bubble/index.html` の紙吹雪(コピー改修)、`common/haptics.js` / `common/highscore.js` / `common/treasure.js` / `common/narration.js`。
- **新規開発が必要な部分**: 出現スケジューラ、CSS transform/clip-path の出現アニメ、**「zzz睡眠」「寝ぼけ伸び」表情が全キャラ現状ゼロ枚**のため最小3種でも6枚前後の GPT Image 2 生成、回帰テスト。
- **クイズ系との差別化**: 判断材料は「起きてる/寝てる」の視覚のみで言語・知識ゼロ。考える時間を与えない即時反応。
- **懸念点と対応方針**:
  - **連打(spam-tap)で go/no-go が無効化される問題が未設計** → maze の no-spam-solve audit の前例に倣い、誤タップペナルティ/クールダウンを必須仕様化しテストケース化。これが最大の宿題。
  - 表情差分「揃っている」は誤認 → まず hedgehog 1キャラでテスト生成→画風一致検証のゲートを設ける。
  - 隠れ場所(屋内家具流用 vs 屋外切り株新規)・グリッド仕様・tier を着手前に確定。
- **推奨度**: **中期検討**(連打対策設計+画風テスト生成が通れば A 昇格候補)。

#### B2. ポノのはたけにっき(毎日みずやり菜園)— 育成・お世話系(お世話枠の代表)

- **レビュー**: revise / diversity 3 / feasibility 3
- **コンセプト概要**: リアル時間で数日かけて植物を育てる生活習慣型ループ。水やり・草取り・虫逃がし・日なた移動の4ジェスチャー。水やりを忘れても枯れず「しおれて待つ」優しい設計。収穫→宝箱→はたけずかん。
- **流用する既存資産**: `assets/images/yard/`(hatake.png/hatake_crop.png 等)、`common/stickers.js` の checkDailyLogin(日付跨ぎ判定の参照実装)、`common/achievements.js`、`common/treasure.js`(choices 付き収穫演出)、`common/encyclopedia.js` + `common/museum-data.js`(plants.json 同型でずかん化)、`common/narration.js` / `common/haptics.js` / `common/cloud-sync.js`。
- **新規開発が必要な部分**: 実時間ベースの成長ステートマシン(**このコードベースに前例のない領域**)、4ジェスチャー実装、植物×成長段階の画像(しおれ差分込みで実態は40〜60枚規模になりうる)、時刻モック可能なテストユーティリティ。
- **クイズ系との差別化**: 正解/不正解の概念自体が存在せず、報酬は「昨日まいた種が今日どうなっているか」という実時間の変化への期待。既存に無い継続動機の型。
- **懸念点と対応方針**:
  - 時刻ベース状態機械が新規領域なのに S〜M 評価は楽観的 → 「植物2種・しおれ差分なし」の最小S版で見積もり直し、段階投入。
  - cloud-sync との2端末マージ戦略が未設計 → 実装前に cloud-sync.js を読みマージ方針(last-write-wins 等)を文書化。
  - 複数日欠損(旅行等)からの復帰パス → 1日ずつ逐次シミュレートで多日ジャンプに耐える設計をテスト先行(TDD を MUST 運用)で確定。
  - 時刻チートは完全防止不可 → 被害限定方針(得られるのは種の解放が早まる程度)を明記。
- **推奨度**: **中期検討**(お世話枠の本命。時刻モック基盤を先に作る)。

#### B3. ぐらぐらシーソーひろば(てんびんバランス)— 物理・構築系

- **レビュー**: revise / diversity 4 / feasibility 2.5
- **コンセプト概要**: シーソーに重さの違う動物・果物を乗せて釣り合わせる。数字を一切見せず、傾きだけがフィードバックの量感・てこの前概念遊び。
- **流用する既存資産**: `assets/images/ocean/` の245種1枚絵(乗せ物候補)、`common/haptics.js` / `common/narration.js` / `common/achievements.js`、`puzzle/assists/_hooks-init.js` の PonoAssistRegister パターン(補助レイヤ設計の参考)。
- **新規開発が必要な部分**: トルク+減衰振動の物理コア(50〜100行)、ドラッグ&ドロップ、乗せ物カタログJSON、タイトル/サムネ絵。実態は6〜7割新規。
- **クイズ系との差別化**: 数値も選択肢も見せず、釣り合う組み合わせが毎回複数ある身体的試行錯誤。
- **懸念点と対応方針**:
  - **「絵の大きさ=重さ」が ocean 素材のバラバラなスケール感と矛盾**(教育的整合性の穴) → 10〜15点の厳選カタログに絞るか、沈み込み量など画像サイズ非依存の重さ表現に変更。着手前に決定。
  - **monster_math 撤退(同系「さんすう前概念」枠での否定的フィードバック)の前例** → フルアート投資前に紙モック/簡易プロトで未就学児テストを必須ゲート化。tier 判断もオーナー確認を先行。
  - てこ(支点距離)機能の v1 スコープ可否を先に確定。滑り落ち演出は非罰的表現(跳ね返り)に変更。
- **推奨度**: **中期検討**(プロトタイプ検証ゲート通過が条件)。

## 5. 統括: ジャンルバランスと着手順

### ジャンルバランス所見

- **本命が立ったジャンル**: 戦略・かけひき(A1)、アクション(A2/B1)、育成(B2)、創作(B4)。この4ジャンルはクイズ構造からの距離が遠く、既存タイトルとの重複も小さい。
- **既存資産と競合したジャンル**: リズム(oto と全面重複)、探索サンドボックス(aquarium/zukan と重複)。このジャンルは「新規タイトル追加」ではなく「既存タイトルの拡張」として扱う方が投資効率が高い、というのが今回のレビュー群から得られた最大の学びである。
- **検証ゲートが必要なジャンル**: 物理・構築(B3/B8)は monster_math 撤退・bowling 廃止という2つの先行撤退事例に隣接しており、フルアート投資前の低コストプロトタイプ検証を必須とする。
- **横断的な教訓**: 21件中かなりの案で「流用できる」とされた資産(表情差分・parallax 実運用・drag 共通モジュール等)が実在しなかった。今後の企画は reused_systems 記載前に ls/grep での実在確認を必須プロセスとすべきである。

### 推奨着手順(いずれもオーナー承認後)

1. **Phase 1(すぐ)**: A1 どんぐりわけっこ対決 + A2 ぱくぱくキャッチ。前者は差別化×実現性の最良バランス、後者はアクション枠パイロット兼、以降の全ドラッグ系ゲームの土台となる iOS 対応ドラッグ実装の先行投資。両案とも着手前に本書記載の設計確定事項(NPC方式・終了条件・素材生成計上)を消化する。
2. **Phase 2(短中期)**: B1 ひょっこりハイタッチ(連打対策設計+画風テスト生成の2ゲート通過後)、B2 はたけにっき(時刻モック基盤の整備とセット)。ここでアクション・育成の2ジャンルが確立する。
